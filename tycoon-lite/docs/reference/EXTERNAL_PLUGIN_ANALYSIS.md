### Jobs Reborn 분석

| 항목 | 평가 |
|------|------|

### 권장 사항
```
⚠️ 조건부 도입
```

```java
    FarmerExpService.giveExp(player, crop);
    FarmerSellService.calculatePrice(crop, grade);
}
```

## 4. 인챈트 시스템: ExcellentEnchants

### 현재 상태
- 기존 `enhance/` 패키지: 43개 파일
- **11개 커스텀 인챈트** (출혈, 번개강타, 흡혈 등)
- **57개 램프 효과** (완전 고유 시스템)
- **천장 시스템** (200회 = LEGENDARY 확정)

### ExcellentEnchants 분석

| 항목 | 평가 |
|------|------|
| **적합도** | ⭐⭐⭐ (중간) |
| **커스텀 여지** | 🟡 중간 - 설정 기반 |
| **연동 복잡도** | 🔴 높음 - 램프 시스템과 충돌 |
| **비용** | 무료 |

### 권장 사항
```
⚠️ 부분 도입 검토
- 문제: 기존 램프 시스템이 매우 독특하고 깊음
- ExcellentEnchants는 "인챈트"에 집중
- 램프 ≠ 인챈트 (Tycoon에서 별도 시스템)

선택지:
A) 기존 시스템 유지 (권장)
B) ExcellentEnchants로 인챈트만 대체, 램프는 유지
C) 완전 대체 (램프 시스템 포기)
```

### 내 권장
```
A) 기존 시스템 유지
- 램프 57개 효과 + 천장 시스템은 매우 고유함
- 이걸 버리고 범용 플러그인 쓰는 건 손해
- 다만 ExcellentEnchants의 일부 효과 아이디어만 참고 가능
```

## 🔄 재설계 관점 분석 (NEW)

> **핵심 질문**: "기존 설계를 버리고 외부 플러그인에 맞춰 재설계하는 것이 더 나은가?"
> 
> 이 섹션은 각 시스템별로 **유지 vs 재설계**를 깊이 비교합니다.

---

### 재설계 원칙

```
1. 외부 플러그인이 "더 잘하는 것"은 위임
2. 우리만의 "고유한 가치"는 유지
3. 연결점(Glue Code)만 우리가 작성
4. 설계 문서는 참고용으로 보존, 구현은 플러그인 기준
```

---

### A. 직업 시스템: Jobs Reborn vs Tycoon Jobs

#### 비교

| 측면 | Tycoon Jobs (현재) | Jobs Reborn |
|------|-------------------|-------------|
| **철학** | 커스텀 가공 체인 + 깊은 메카닉 | 바닐라 행동 → 보상 |
| **복잡도** | 매우 높음 (113파일) | 중간 (설정 기반) |
| **광부 특징** | 4단계 철, 산화 구리, 정제소 | 캐면 돈/경험치 |
| **확장성** | 코드 수정 필요 | 설정으로 추가 |
| **고유성** | ⭐⭐⭐⭐⭐ | ⭐⭐ |

#### 재설계 시 변화

```
Jobs Reborn 채택 시:
✅ 장점:
- 설정만으로 직업 추가/수정 가능
- 검증된 안정성
- 플레이어가 익숙한 시스템

❌ 포기해야 할 것:
- 4단계 철 가공 체인 (Pig Iron → Steel → Pure Iron → Alloy Steel)
- 산화 구리 시스템 (산화 진행 규칙, 2×/4× 가치)
- 정제소 계층 구조
- 원유/석영 가공
- 레벨 0 앵커 기반 수익 설계
```

#### 결론

```
❌ 재설계 비권장
- Tycoon의 광부 시스템은 "서버의 핵심 정체성"
- Jobs Reborn은 "바닐라+ 간편함"에 적합
- 우리 서버는 "복잡한 가공 체인"이 차별점

만약 재설계한다면:
- Jobs Reborn의 "기본 경험치/레벨" 시스템만 차용
- 가공 체인은 별도 시스템으로 구현
- 하지만 이건 더 복잡해짐 → 비추천
```

### B. 농사 시스템: CustomCrops vs Tycoon Farmer

#### 비교

| 측면 | Tycoon Farmer (현재) | CustomCrops |
|------|---------------------|-------------|
| **커스텀 작물** | 코드로 정의 | YAML로 정의 |
| **성장 메카닉** | 스프링클러, 비료, 물주기 | 계절, 온도, 습도, 비료 |
| **텍스처** | 리소스팩 직접 관리 | 내장 지원 |
| **확장성** | 코드 수정 | 설정으로 추가 |
| **고유성** | ⭐⭐⭐ | ⭐⭐ (범용) |

#### 재설계 시 변화

```
CustomCrops 채택 시:
✅ 장점:
- 작물 추가가 매우 쉬움 (YAML)
- 계절 시스템 기본 제공
- 텍스처/모델 관리 편함
- 플레이어 친화적 UX

⚠️ 조정 필요:
- 스프링클러 시스템 → CustomCrops 방식으로 재구현 또는 통합
- 비료 시스템 → CustomCrops에 맡기기
- 물주기 → CustomCrops 방식 채택

✅ 유지 가능:
- 등급/경험치/레벨 (Tycoon Farmer 핵심)
- 판매 가격 계산 (경제 시스템)
- 직업 보너스
```

#### 결론

```
✅ 재설계 권장
- CustomCrops가 "작물 정의/성장"을 더 잘함
- Tycoon은 "경제/등급/보상"에 집중

재설계 방향:
┌─────────────────────┐     ┌─────────────────────┐
│   CustomCrops       │ ──→ │   Tycoon Farmer     │
│ - 작물 정의         │     │ - 수확 이벤트 감지  │
│ - 성장 메카닉       │     │ - 경험치 계산       │
│ - 계절/비료/물      │     │ - 등급/보너스 적용  │
│ - 텍스처/모델       │     │ - 판매가 계산       │
└─────────────────────┘     └─────────────────────┘

기존 설계 문서 처리:
- FarmerWaterSystem.java → CustomCrops로 대체
- FarmlandLimitSystem.java → CustomCrops 연동으로 재작성
- 등급/경험치 로직 → 유지
```

### C. 낚시 시스템: PyroFishing Pro vs Tycoon Fisher

#### 비교

| 측면 | Tycoon Fisher (현재) | PyroFishing Pro |
|------|---------------------|-----------------|
| **낚시 과정** | 바닐라 기반 | 미니게임 |
| **물고기 정의** | 코드로 정의 | YAML로 정의 |
| **AFK 방지** | 자체 구현 | 내장 |
| **미끼 시스템** | 자체 구현 | 내장 + 확장 |
| **토너먼트** | 없음 | 내장 |
| **고유성** | ⭐⭐⭐ | ⭐⭐⭐⭐ (기능 풍부) |

#### 재설계 시 변화

```
PyroFishing Pro 채택 시:
✅ 장점:
- 낚시 미니게임 (플레이어 참여도 ↑)
- 토너먼트 시스템 기본 제공
- 커스텀 물고기 YAML 정의
- 더 다양한 컨텐츠

⚠️ 조정 필요:
- 피티 시스템 → PyroFishing 방식 확인 필요
- AFK 방지 → PyroFishing 내장 사용
- 미끼 시스템 → PyroFishing으로 대체

✅ 유지 가능:
- 경험치/레벨 (Tycoon Fisher)
- 판매 가격/등급 (경제 시스템)
- 직업 보너스
```

#### 결론

```
✅ 재설계 권장
- PyroFishing이 "낚시 컨텐츠"를 훨씬 잘함
- 미니게임이 플레이어 경험을 크게 개선

재설계 방향:
┌─────────────────────┐     ┌─────────────────────┐
│   PyroFishing Pro   │ ──→ │   Tycoon Fisher     │
│ - 낚시 미니게임     │     │ - 낚시 결과 감지    │
│ - 물고기 정의       │     │ - 경험치 계산       │
│ - 미끼/장비         │     │ - 등급/보너스 적용  │
│ - 토너먼트          │     │ - 판매가 계산       │
└─────────────────────┘     └─────────────────────┘

기존 설계 문서 처리:
- FisherBaitSystem.java → PyroFishing으로 대체
- FisherAfkSystem.java → PyroFishing 내장 사용
- 피티 시스템 → 확인 후 결정 (유지 or 대체)
```

### D. 인챈트/램프 시스템: ExcellentEnchants vs Tycoon Enhance

#### 비교

| 측면 | Tycoon Enhance (현재) | ExcellentEnchants |
|------|----------------------|-------------------|
| **인챈트 수** | 11개 (커스텀) | 50+ (범용) |
| **램프 시스템** | 57개 효과, 고유 | 없음 |
| **천장 시스템** | 200회 피티 | 없음 |
| **설계 철학** | 램프 ≠ 인챈트 분리 | 인챈트만 |
| **고유성** | ⭐⭐⭐⭐⭐ | ⭐⭐ (범용) |

#### 재설계 시 변화

```
ExcellentEnchants 채택 시:
✅ 장점:
- 50+ 인챈트 즉시 사용
- 검증된 밸런스
- 설정으로 커스터마이징

❌ 포기해야 할 것:
- 램프 시스템 전체 (9개 타입, 57개 효과)
- 천장(피티) 시스템
- 램프 희귀도 체계
- "램프 ≠ 인챈트" 분리 컨셉
```

#### 결론

```
❌ 재설계 비권장 (조건부)
- 램프 시스템이 Tycoon의 핵심 차별점
- 천장 시스템도 고유한 가치

하지만 만약 재설계한다면:
선택지 A: 램프 포기
- ExcellentEnchants만 사용
- 간단하지만 서버 정체성 손실

선택지 B: 하이브리드
- ExcellentEnchants = 바닐라 인챈트 대체
- 램프 시스템 = 별도 유지
- 두 시스템 공존 (가능하지만 복잡)

선택지 C: 유지 (권장)
- 기존 램프 + 인챈트 시스템 유지
- ExcellentEnchants 아이디어만 참고
```

### G. Tier 2 직업: 외부 플러그인 기반 재설계

> Tier 2 직업은 아직 DROP 상태이므로, 처음부터 외부 플러그인 기반으로 설계 가능

#### 장인 (Artisan): ExecutableItems 💰

```
✅ 재설계 권장
- 기존 설계: 장인의 제작대, 커스텀 레시피
- ExecutableItems: 커스텀 아이템 + 인터랙션 정의

재설계 방향:
- ExecutableItems로 "설비" 정의
- Tycoon Artisan = 경험치/등급/레시피 관리
```

#### 약초꾼 (Herbalist): ItemsAdder 💰

```
✅ 재설계 권장
- 기존 설계: 커스텀 허브, 포션
- ItemsAdder: 커스텀 블록/아이템 정의

재설계 방향:
- ItemsAdder로 "허브/포션" 정의
- Tycoon Herbalist = 채집/조합 로직
```

#### 공학자 (Engineer): MechanicsCore 💰

```
✅ 재설계 권장
- 기존 설계: 기계, 자동화
- MechanicsCore: 기계 시스템 프레임워크

재설계 방향:
- MechanicsCore로 "기계" 동작 정의
- Tycoon Engineer = 설계도/업그레이드/경험치
```

#### 요리사 (Chef): CustomCrops + ItemsAdder

```
⚠️ 조건부 재설계
- 기존 설계: 요리 레시피, 주방
- CustomCrops: 농작물 가공 확장
- ItemsAdder: 음식 아이템 정의

재설계 방향:
- ItemsAdder로 "음식 아이템" 정의
- CustomCrops 연동으로 재료 확보
- Tycoon Chef = 레시피/조리/버프 관리
```

### 🎯 최종 재설계 전략

#### 1. 완전 대체 (기존 코드 버림)
```
- town/ → Lands
- dungeon/ → MythicDungeons
```

#### 2. 인프라 위임 + 로직 유지
```
- job/farmer/ 작물 정의 → CustomCrops
                경험치/등급 → Tycoon 유지

- job/fisher/ 낚시 과정 → PyroFishing
              경험치/등급 → Tycoon 유지

- item/ 아이템 정의 → Oraxen
        가공 체인 → Tycoon 유지
```

#### 3. 완전 유지 (외부 플러그인 안 씀)
```
- job/miner/ 가공 체인 전체
- enhance/ 램프 시스템 전체
- codex/, achievement/ 도감/업적 전체
- economy/, shop/ 경제 전체
```

#### 4. 처음부터 플러그인 기반 (Tier 2)
```
- job/artisan/ → ExecutableItems 기반
- job/herbalist/ → ItemsAdder 기반
- job/engineer/ → MechanicsCore 기반
- job/chef/ → ItemsAdder + CustomCrops 기반
```
